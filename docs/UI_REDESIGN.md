# UI再設計メモ

更新日: 2026-07-18

## 対象

- 20〜30歳の参加者
- スマートフォン操作に慣れていない人を含む
- 最小対応幅320px

## 操作原則

- 注文は「1 選ぶ → 2 確認 → 3 送る」の順に進める。
- 送信前に注文者・送信先・商品・サイズ・個数・合計を表示する。
- 主な操作面は44px以上、本文は16px以上を基準にする。
- ブラウザの拡大を禁止しない。
- 成功・エラー・空・読込み中を見た目と文で区別する。
- エラー時は成功表示にせず、再試行前に注文履歴を確認するよう案内する。
- festival ticketを基調とした赤・藍・金の見た目を維持し、Discord色はログインボタンだけに使う。

## ログイン

- 所属グループやDiscord名の手入力を廃止し、`Discordでログイン`の1ボタンにした。
- 管理者が本人確認済みDiscord User IDを事前登録し、登録済みアカウントだけを許可する。
- 未登録アカウントは理由を明示してログイン画面へ戻し、未知のDiscord情報を保存しない。
- IDそのもの・表示名・プロフィール・メッセージは保存せず、ログイン照合用HMACだけを保存することを明記した。
- 初回の未ログインを「session期限切れ」と誤表示しない。

## 実装済み

- Discord OAuth session復元・server logout
- member / manager / admin別navigation
- 注文cart・最終確認・冪等送信
- カート追加を一時toastで通知し、画面下部のカートを常時固定表示
- 深夜料金の注意をJST 21:30〜翌5:59だけ注文画面へ表示（会計画面では照合漏れ防止のため常時表示）
- 自分の注文履歴
- 管理者が追加した注文の追加元を本人履歴へ表示
- managerの自グループ注文集約・数量変更・伝達済み
- managerの自グループ会計
- non-adminには宴会コースをAPI・画面の両方で非表示
- adminの参加者・ログイン許可・menu・代理注文・統計・監査ログ
- 宴会コースの代理注文は即「注文済み」として会計へ加算
- 管理者が事前追加した注文だけ、監査付きで訂正可能
- 参加者を1人ずつ、または名前とDiscord User IDの2一覧から最大100人まとめて事前登録
- 参加者の安全な利用停止（ログインを止め、注文・会計・操作履歴は保持）
- 消える件数と残る件数を先に示す、再ログイン・復元地点確認付きの開催データ初期化
- privacy coverと自動logout案内
- PWA更新案内

## 意図的に停止している機能

- Web Push通知
- 利用者本人が送った注文の取消（管理者が事前追加した注文の訂正だけ利用可）
- 参加者の物理削除

未完成のボタンは置きません。取消は監査付きAPIが完成するまで、管理者へ直接伝える案内だけを表示します。

## 管理者の参加者操作

- 一括追加は「名前とDiscord User IDを同じ順番で1人ずつ改行 → 全員のグループと権限を選択 → 組み合わせを確認 → 全員を追加」の順にする。見出し行や区切り文字は求めない。
- 名前とDiscord User IDの人数が違う場合や先頭・途中に空行がある場合は、別人のIDを結び付けずに止める。
- 1行でも入力エラーや登録済みDiscordアカウントがあれば、誰も追加しない。
- 確認一覧ではDiscord User IDの全文を表示せず、末尾4桁だけを表示する。
- 入力中のDiscord User IDはlocalStorageやsessionStorageへ保存しない。
- 参加者の利用停止確認では、ログインできなくなること、現在の端末もログアウトすること、過去の履歴は残ることを実行ボタンの直前に表示する。
- 画面では物理削除と誤解される「削除」を使わず、「利用停止」に統一する。唯一の管理者には「管理者は利用停止できません」と理由を表示する。
- 開催データ初期化は管理者・メニュー・操作履歴を保持し、削除した参加者と過去の操作履歴との紐づけは解除されることを明記する。実行前後の結果は画面内の読み上げ領域でも通知する。
